# 1. ВВЕДЕНИЕ

## 1.1. Описание назначения продукта и причин его создания

**hodotaev_praktika** представляет собой специализированное десктопное приложение, разработанное для автоматизации управления партнерской сетью производственной компании «Мастер Пол», занимающейся производством и реализацией напольных покрытий. Продукт предназначен для менеджеров по работе с партнерами, аналитиков и руководителей отдела продаж, нуждающихся в систематизации информации о партнерах, контроле их деятельности и расчете индивидуальных условий сотрудничества.

### Причины создания продукта:

**1. Отсутствие доступных специализированных решений.** Существующие CRM-системы (1С:CRM, Salesforce, Bitrix24) обладают избыточным функционалом для задач управления партнерской сетью, требуют значительных затрат на лицензирование, внедрение и обучение персонала. Специализированные модули для работы с дистрибьюторами в составе ERP-систем (SAP, Oracle) имеют высокую стоимость и сложность настройки, что делает их недоступными для средних производственных компаний.

**2. Неэффективность табличных процессоров.** Традиционный подход к учету партнеров через Microsoft Excel или Google Sheets сопряжен с многочисленными проблемами: отсутствие контроля целостности данных (возможность дублирования записей, некорректные ссылки на типы партнеров), риск потери данных при совместной работе, отсутствие автоматизации расчета скидок и аналитики, низкая скорость обработки при объеме данных свыше 2000 записей.

**3. Потребность в специализированном учете партнерской деятельности.** Управление партнерами производственной компании обладает спецификой: необходимость ведения истории продаж для расчета индивидуальных скидок, классификация партнеров по типам (оптовые, розничные, интернет-магазины, дистрибьюторы), контроль рейтинга надежности, отслеживание истории изменений условий сотрудничества. Универсальные CRM-системы не учитывают данную специфику без дорогостоящей доработки.

**4. Импортозамещение и технологическая независимость.** В условиях санкционного давления и ухода западных вендоров (Salesforce, Oracle, SAP) с российского рынка, разработка отечественного программного обеспечения на открытых технологиях (.NET 8, PostgreSQL, Entity Framework Core) обеспечивает технологический суверенитет, независимость от иностранных поставщиков и возможность самостоятельной поддержки и развития системы.

**5. Образовательная и исследовательская цель.** Проект служит практической апробацией знаний в области проектирования архитектуры программных систем, применения паттернов проектирования (MVP, Dependency Injection, Repository), работы с современными ORM (Entity Framework Core 8), создания графических интерфейсов на платформе WPF, проектирования реляционных баз данных в PostgreSQL с соблюдением нормальных форм.

### Область применения:

- Просмотр списка партнеров с детальной информацией;
- Добавление и редактирование данных о партнере;
- Ведение справочника типов партнеров;
- Расчет индивидуальной скидки партнера на основе объема продаж;
- Просмотр истории реализации продукции партнером;
- Контроль рейтинга партнеров;
- Удаление партнеров с проверкой ссылочной целостности;
- Поиск и фильтрация партнеров по различным критериям.

Продукт ориентирован на менеджеров по работе с партнерами, руководителей отдела продаж, аналитиков и администраторов системы, не обладающих специальной технической подготовкой. Интерфейс приложения спроектирован с учетом принципов интуитивной понятности, минимизации времени обучения персонала и обеспечения комфортной длительной работы.

### Актуальность разработки

Актуальность разработки подтверждается тенденциями развития российского рынка напольных покрытий: по данным агентства «РБК.Исследования рынков», объем рынка напольных покрытий в Российской Федерации в 2024-2025 годах демонстрирует устойчивый рост на 15-20% ежегодно. Увеличение числа партнеров и расширение географии продаж формирует спрос на доступные, легкие в развертывании и использовании инструменты автоматизации управления партнерской сетью.

Переход производственных компаний на прямую работу с партнерской сетью (минуя крупных дистрибьюторов) требует детального контроля каждого партнера, анализа его вклада в общий объем продаж и предоставления индивидуальных условий сотрудничества. Существующие универсальные решения не обеспечивают необходимой гибкости и специализации для данных задач.

Таким образом, создание **hodotaev_praktika** решает актуальную прикладную задачу автоматизации управления партнерами производственной компании, а также реализует учебно-исследовательские цели в рамках выполнения курсовой работы по дисциплине «Технология разработки программного обеспечения».

## 1.2. Требования к продукту

### 1.2.1. Функциональные требования

**Требования к управлению партнерами:**

1. Система должна обеспечивать просмотр списка всех партнеров с отображением ключевой информации: наименование компании, тип партнера, рейтинг, контактные данные, общий объем продаж, размер скидки.

2. Система должна предоставлять возможность добавления нового партнера с вводом следующих данных:
   - Наименование компании (обязательное поле);
   - Тип партнера (выбор из справочника);
   - Юридический адрес;
   - ИНН;
   - ФИО директора;
   - Телефон;
   - Email;
   - Рейтинг (неотрицательное целое число).

3. Система должна предоставлять возможность редактирования данных существующего партнера с загрузкой актуальных значений из базы данных.

4. Система должна предоставлять возможность удаления партнера при условии отсутствия связанных записей о продажах (защита от каскадного удаления).

5. Система должна обеспечивать валидацию вводимых данных:
   - Проверка обязательности заполнения наименования компании;
   - Проверка выбора типа партнера;
   - Проверка неотрицательности рейтинга;
   - Проверка корректности формата email;
   - Проверка уникальности ИНН.

**Требования к расчету скидок:**

6. Система должна автоматически рассчитывать размер скидки для каждого партнера на основании общего объема реализованной продукции за весь период работы по следующим правилам:
   - Объем продаж до 10 000 рублей — скидка 0%;
   - Объем продаж от 10 000 до 50 000 рублей — скидка 5%;
   - Объем продаж от 50 000 до 300 000 рублей — скидка 10%;
   - Объем продаж свыше 300 000 рублей — скидка 15%.

7. Система должна отображать рассчитанную скидку в общем списке партнеров и в карточке партнера.

**Требования к истории продаж:**

8. Система должна предоставлять возможность просмотра истории реализации продукции партнером с отображением:
   - Даты продажи;
   - Наименования продукции;
   - Артикула продукции;
   - Количества проданных единиц;
   - Цены за единицу;
   - Общей суммы продажи.

9. Система должна обеспечивать возможность открытия истории продаж в отдельном окне с подробной информацией о партнере.

**Требования к справочникам:**

10. Система должна поддерживать справочник типов партнеров с возможностью:
    - Просмотра списка типов;
    - Добавления новых типов;
    - Редактирования существующих типов;
    - Запрета удаления типов, используемых партнерами.

### 1.2.2. Нефункциональные требования

**Требования к производительности:**

1. Время загрузки списка партнеров (до 1000 записей) — не более 2 секунд.
2. Время открытия истории продаж партнера — не более 1,5 секунд.
3. Время выполнения операции добавления/редактирования партнера — не более 1 секунды.
4. Потребление оперативной памяти — не более 250 МБ.

**Требования к надежности:**

5. Система должна обеспечивать сохранность данных при аварийном завершении работы (транзакционность операций).
6. Система должна предотвращать удаление данных, на которые существуют ссылки (защита ссылочной целостности).
7. Система должна корректно обрабатывать ошибки подключения к базе данных с информированием пользователя.

**Требования к удобству использования:**

8. Интерфейс системы должен быть выполнен в едином стиле.
9. Все окна должны иметь информативные заголовки.
10. Сообщения об ошибках должны содержать понятный текст с рекомендациями по устранению.
11. Система должна подтверждать выполнение успешных операций (добавление, редактирование, удаление).
12. Система должна запрашивать подтверждение перед выполнением необратимых операций (удаление).

**Требования к совместимости:**

13. Система должна функционировать под управлением ОС Windows 10/11 (64-bit).
14. Система должна работать с СУБД PostgreSQL версии 14 и выше.
15. Система должна использовать .NET 8.0 Runtime.

# 2. ПРОЕКТИРОВАНИЕ

## 2.1. Эскизный проект (концепция)

### 2.1.1. Название проекта

**Полное наименование:** «Автоматизированная подсистема для работы с партнерами компании hodotaev_praktika»

**Краткое наименование:** hodotaev_praktika

**Версия:** 1.0.0

**Кодовое имя:** «HP-Partners-2026»

### 2.1.2. Цели проекта

**Стратегическая цель:** Разработать законченное, готовое к эксплуатации десктопное приложение для автоматизации управления партнерами производственной компании, обеспечивающее ведение единого реестра партнеров, расчет индивидуальных скидок на основе истории продаж, просмотр детальной информации о реализации продукции.

**Тактические цели:**

1. Создать модульную архитектуру приложения с разделением на библиотеку бизнес-логики (hodotaev_library) и клиентский интерфейс (hodotaev_wpf), обеспечивающую возможность независимого тестирования и повторного использования кода.

2. Реализовать полнофункциональную систему управления партнерами (CRUD-операции) с валидацией вводимых данных и защитой от некорректных операций.

3. Разработать подсистему расчета индивидуальных скидок на основе анализа истории продаж партнера.

4. Обеспечить механизмы просмотра истории реализации продукции с детализацией по каждой продаже.

5. Спроектировать реляционную базу данных в СУБД PostgreSQL с поддержкой ссылочной целостности, соблюдением третьей нормальной формы, индексацией поисковых полей.

6. Реализовать пользовательский интерфейс в современном стиле с использованием карточек партнеров, соответствующий требованиям методической документации.

7. Подготовить полный комплект проектной и эксплуатационной документации.

**Основные результаты (продукты):**

3. **Техническая документация:**

### 2.1.4. Допущения и ограничения

**Допущения (принятые при проектировании):**

1. Пользователь приложения обладает базовыми навыками работы с персональным компьютером на уровне уверенного пользователя Windows.

2. Целевая операционная система — Windows 10/11 (64-bit). Поддержка других ОС (Linux, macOS) не предполагается в рамках текущей версии.

3. Сервер баз данных PostgreSQL развертывается локально либо доступен по сети. Администрирование сервера БД входит в компетенцию заказчика.

4. Предполагается однопользовательский режим работы. Механизмы блокировок при конкурентном доступе не реализуются.

5. Начальные данные (справочник типов партнеров) создаются автоматически при инициализации базы данных.

6. Ввод данных о продажах осуществляется отдельно. В рамках данной подсистемы реализуется только просмотр истории продаж.

2. **Функциональные:**
   - Максимальный объем базы данных — до 50 000 партнеров (без деградации производительности);
   - Отсутствие многопользовательского режима и ролевой модели в версии 1.0;

## 2.3. Проектирование базы данных

### 2.3.1. Схема базы данных

База данных спроектирована в СУБД PostgreSQL с соблюдением третьей нормальной формы (3NF) и обеспечением ссылочной целостности.

**Схема: app**

Все таблицы размещены в схеме `app`, владельцем которой является роль `app`. Такое разделение обеспечивает логическую изоляцию объектов приложения от системных объектов PostgreSQL.

**Таблицы:**

1. **hodotaev_partner_types** — справочник типов партнеров
   - Первичный ключ: partner_type_id
   - Уникальное ограничение: type_name

2. **hodotaev_partners** — основная таблица партнеров
   - Первичный ключ: partner_id
   - Внешний ключ: partner_type_id → hodotaev_partner_types
   - Проверочное ограничение: rating >= 0

3. **hodotaev_products** — справочник продукции
   - Первичный ключ: product_id
   - Уникальное ограничение: article

4. **hodotaev_sales** — история продаж
   - Первичный ключ: sale_id
   - Внешние ключи:
     - partner_id → hodotaev_partners (ON DELETE RESTRICT)
     - product_id → hodotaev_products (ON DELETE RESTRICT)
   - Проверочные ограничения: quantity > 0, sale_price >= 0

5. **hodotaev_partner_rating_history** — история изменений рейтинга
   - Первичный ключ: history_id
   - Внешний ключ: partner_id → hodotaev_partners (ON DELETE CASCADE)

////////////СЮДА ДИАГРАММУ 4 (ER-диаграмма базы данных)//////

### 2.3.2. Обеспечение третьей нормальной формы

**Первая нормальная форма (1NF):**
- Все поля таблиц содержат атомарные значения;
- Отсутствуют повторяющиеся группы полей;
- Каждая запись уникально идентифицируется первичным ключом.

**Вторая нормальная форма (2NF):**
- Все неключевые поля полностью зависят от первичного ключа;
- Отсутствуют частичные зависимости от части составного ключа.

**Третья нормальная форма (3NF):**
- Отсутствуют транзитивные зависимости неключевых полей;
- Все справочные данные вынесены в отдельные таблицы (типы партнеров, продукция).

### 2.3.3. Ссылочная целостность

Ссылочная целостность обеспечивается следующими механизмами:

1. **Внешние ключи (Foreign Keys):**
   - Запрет удаления типа партнера, если существуют партнеры данного типа (RESTRICT);
   - Запрет удаления партнера, если существуют записи о продажах (RESTRICT);
   - Каскадное удаление истории рейтинга при удалении партнера (CASCADE).

2. **Проверочные ограничения (Check Constraints):**
   - rating >= 0 — рейтинг не может быть отрицательным;
   - quantity > 0 — количество в продаже должно быть положительным;
   - sale_price >= 0 — цена продажи не может быть отрицательной;
   - min_price >= 0 — минимальная цена не может быть отрицательной.

3. **Уникальные ограничения (Unique Constraints):**
   - type_name — уникальность наименования типа партнера;
   - article — уникальность артикула продукции.

### 2.3.4. Индексы

Для оптимизации производительности созданы следующие индексы:

| Таблица | Поле | Назначение |
|---------|------|------------|
| hodotaev_partners | partner_type_id | Ускорение поиска по типу партнера |
| hodotaev_sales | partner_id | Ускорение выборки истории продаж партнера |
| hodotaev_sales | product_id | Ускорение выборки продаж по продукту |
| hodotaev_sales | sale_date | Ускорение выборки по периоду |
| hodotaev_partner_rating_history | partner_id | Ускорение выборки истории рейтинга |

### 2.3.5. Функции и триггеры

**Функция hodotaev_calculate_discount:**
- Назначение: расчет размера скидки партнера на основе объема продаж;
- Входной параметр: partner_id — идентификатор партнера;
- Возвращаемое значение: DECIMAL(5,2) — процент скидки;
- Логика:
  - sales < 10000 → 0%;
  - 10000 ≤ sales < 50000 → 5%;
  - 50000 ≤ sales < 300000 → 10%;
  - sales ≥ 300000 → 15%.

**Функция hodotaev_update_updated_at_column:**
- Назначение: автоматическое обновление поля updated_at при изменении записи;
- Тип: триггерная функция;
- Триггер: trg_partners_update_updated_at (BEFORE UPDATE на hodotaev_partners).

**Представление hodotaev_partners_summary:**
- Назначение: получение сводной информации о партнерах с расчетом общего объема продаж;
- Поля: partner_id, company_name, partner_type, rating, phone, email, total_sales_amount.

## 2.5. Обработка исключительных ситуаций

### 2.5.1. Типы исключений

В приложении реализована обработка следующих типов исключений:

**1. Исключения базы данных (DbException):**
- Причина: недоступность сервера PostgreSQL, ошибки подключения;
- Действие: информирование пользователя с рекомендацией проверить подключение к БД.

**2. Исключения валидации (ValidationException):**
- Причина: некорректные данные, введенные пользователем;
- Действие: отображение сообщения с указанием конкретной ошибки.

**3. Исключения целостности данных (InvalidOperationException):**
- Причина: попытка удаления партнера с активными продажами;
- Действие: информирование о невозможности удаления с указанием причины.

**4. Общие исключения (Exception):**
- Причина: непредвиденные ошибки;
- Действие: отображение общего сообщения об ошибке с текстом исключения.

# 5. РУКОВОДСТВО ПОЛЬЗОВАТЕЛЯ

## 5.1. Установка и запуск

**Требования к системе:**
- ОС: Windows 10/11 (64-bit);
- .NET 8.0 Runtime;
- PostgreSQL 14+ (локально или по сети);
- Оперативная память: не менее 4 ГБ;
- Свободное место на диске: не менее 500 МБ.

**Установка:**

1. Установите PostgreSQL с официального сайта: https://www.postgresql.org/download/
2. Выполните скрипт создания базы данных:
   ```
   psql -U postgres -f database/hodotaev_praktika.sql
   ```
3. Скопируйте файлы приложения из папки publish в желаемую директорию.
4. Запустите файл hodotaev_wpf.exe.

**Первый запуск:**

При первом запуске приложение попытается подключиться к базе данных. При успешном подключении отобразится главный экран со списком партнеров.

## 5.2. Работа со списком партнеров

**Просмотр списка:**

При запуске приложения автоматически загружается список всех партнеров из базы данных. Список отображается в левой части окна в виде прокручиваемого списка.

**Выбор партнера:**

Кликните по записи в списке для выделения. Информация о выбранном партнере отобразится в карточке справа.

**Быстрое редактирование:**

Двойной клик по записи в списке открывает окно редактирования партнера.

## 5.3. Добавление партнера

1. Нажмите кнопку «Добавить» на панели инструментов.
2. В открывшемся окне заполните поля:
   - **Наименование компании** (обязательно) — полное наименование организации;
   - **Тип партнера** (обязательно) — выберите из выпадающего списка;
   - **Рейтинг** — введите неотрицательное целое число;
   - **Юридический адрес** — адрес регистрации организации;
   - **ИНН** — идентификационный номер налогоплательщика;
   - **ФИО директора** — фамилия, имя, отчество руководителя;
   - **Телефон** — контактный номер;
   - **Email** — адрес электронной почты.
3. Нажмите кнопку «Сохранить».
4. После успешного сохранения окно закроется, список партнеров обновится.

## 5.4. Редактирование партнера

1. Выберите партнера в списке.
2. Нажмите кнопку «Редактировать» или выполните двойной клик по записи.
3. Измените необходимые поля.
4. Нажмите кнопку «Сохранить».
5. После успешного сохранения окно закроется, список партнеров обновится.

## 5.5. Удаление партнера

1. Выберите партнера в списке.
2. Нажмите кнопку «Удалить».
3. Подтвердите удаление в диалоговом окне.
4. Партнер будет удален из базы данных, список обновится.

**Важно:** Удаление партнера невозможно, если существуют записи о продажах данного партнера. В этом случае отобразится сообщение об ошибке.

## 5.6. Просмотр истории продаж

1. Выберите партнера в списке.
2. Нажмите кнопку «История продаж».
3. В открывшемся окне отобразится таблица всех продаж партнера.
4. Для закрытия окна нажмите кнопку «Закрыть».

## 5.7. Обновление списка

Для принудительного обновления списка партнеров нажмите кнопку «Обновить» на панели инструментов.

## 5.8. Выход из приложения

Для выхода из приложения нажмите кнопку «Выход» или закройте окно стандартным способом.

4. Проверьте создание схемы app и таблиц.

2. Проверьте параметры подключения в App.xaml.cs.
3. Убедитесь, что база данных hodotaev_praktika существует.
4. Проверьте права доступа роли app.

**Проблема:** Данные не загружаются в главный экран.

**Решение:**
1. Проверьте наличие данных в таблицах.
2. Выполните скрипт инициализации начальных данных.
3. Проверьте логи приложения.

**Проблема:** Ошибка при удалении партнера.

**Решение:**
1. Проверьте наличие записей о продажах партнера.
